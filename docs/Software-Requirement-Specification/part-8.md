## Part 8 - Non-Functional Requirements, Performance, Scalability, Deployment and Final Acceptance Criteria

**Project Name:** AME AI Tutor

**Document ID:** SRS-01-Part-8

# 1. Non-Functional Requirements Overview

The AME AI Tutor system should be designed to be:

- Reliable
- Secure
- Maintainable
- Scalable
- Performant
- Available
- Easy to use

The system should be capable of supporting the Version 1 requirements defined in Parts 1 to 7.

---

# 2. Performance Requirements

## NFR-001 - General Application Performance

The application should provide a responsive user experience under normal operating conditions.

Normal application operations should respond within a reasonable amount of time.

Operations that require long-running processing should be handled asynchronously.

Examples:

- PDF Processing
- OCR
- Embedding Generation
- Large Document Processing
- AI Quiz Generation

These operations should not block the user's browser indefinitely.

---

# 3. API Response Performance

For normal API operations that do not require heavy AI processing, the backend should aim to respond quickly.

Examples:

- User Profile
- Dashboard Data
- PDF List
- PDF Metadata
- Study Progress
- Quiz History

The target response time should generally be within:

1-2 seconds

under normal system load.

Actual production performance may vary depending on:

- Server Load
- Network Conditions
- Database Load
- Storage Performance

---

# 4. AI Response Performance

AI-generated operations may take longer than normal API requests.

Examples:

- Study Explanation
- AI Question Answering
- Quiz Generation
- Text Answer Evaluation
- Voice Answer Evaluation

The system should provide a loading state while waiting for AI processing.

Example:

> AI is preparing your answer...

The user should not assume that the application has failed while AI processing is running.

---

# 5. PDF Processing Performance

PDF processing time will depend on:

- PDF Size
- Number of Pages
- Text Complexity
- Number of Images
- OCR Requirement
- Number of Tables
- Document Quality

The system should process PDFs asynchronously.

The user should be able to see the processing status.

---

# 6. Maximum PDF Size

The maximum supported PDF size for Version 1 is:

100 MB

If the uploaded PDF exceeds 100 MB, the system should reject the upload.

The user should be shown a clear message:

> This PDF is larger than 100 MB. Please split the PDF into multiple smaller parts and upload them separately.

The system should not silently truncate or partially process an oversized PDF.

---

Pending Final Infrastructure Decision

The exact limit should be configurable from the backend.

The system should be able to enforce a maximum PDF count per user.

If the user reaches the configured limit, the system should display an appropriate message.

---

# 8. Maximum Concurrent Uploads

The maximum number of simultaneous PDF uploads per user for Version 1 is:

Pending Final Infrastructure Decision

The system should prevent excessive simultaneous uploads from a single user.

The exact limit should be configurable.

---

# 9. Concurrent AI Requests

The system should control the number of simultaneous AI requests generated by a user.

This is required to:

- Protect AI API limits
- Prevent abuse
- Reduce unnecessary costs
- Maintain system stability

The exact limit will be defined during technical implementation.

# 10. Scalability

## NFR-002 - Horizontal Scalability

The application architecture should allow individual services to scale independently where required.

Potentially scalable components include:

- Laravel API
- Python AI Service
- Background Workers
- Database
- Storage

The architecture should avoid unnecessary tight coupling between these components.

---

# 11. Stateless Application Services

The Laravel API and Python AI Service should preferably be designed to be stateless where practical.

Persistent application state should be stored in appropriate infrastructure such as:

- PostgreSQL
- Cloudflare R2
- Redis, if required

This allows multiple application instances to run simultaneously.

---

# 12. Multiple Backend Instances

The system architecture should support running multiple Laravel application instances if the number of users increases.

Example:

User Requests

↓

Load Balancer

↓

Laravel Instance 1

Laravel Instance 2

Laravel Instance 3

↓

Shared Database / Storage

---

# 13. Multiple AI Service Instances

The Python AI Service should be capable of scaling horizontally if AI processing demand increases.

Example:

AI Requests

↓

AI Service Queue

↓

Python AI Worker 1

Python AI Worker 2

Python AI Worker 3

---

# 14. Background Processing

Long-running operations should be processed using background jobs where appropriate.

Examples:

- PDF Processing
- OCR
- Image Extraction
- Table Processing
- Chunking
- Embedding Generation
- Large Quiz Generation

The main API should not remain blocked while these operations are running.

---

# 15. Job Queue

The system should support a background job queue for long-running tasks.

The queue should allow:

- Job Creation
- Job Processing
- Job Retry
- Job Failure Tracking

The exact queue technology will be finalized in the Technical Architecture document.

---

# 16. Failed Job Handling

If a background job fails, the system should:

- Record the failure.
- Log the error.
- Update the relevant resource status.
- Allow retry where appropriate.

Example:

PDF Processing

↓

Failed

↓

PDF Status = Failed

↓

User Can Retry

---

# 17. Retry Policy

Temporary failures may be retried automatically.

Examples:

- Temporary Network Failure
- Temporary AI Provider Error
- Temporary Storage Error

The system should avoid unlimited retries.

A maximum retry count should be configured.

---

# 18. Database Performance

PostgreSQL should be properly indexed for frequently queried data.

Potential indexes may be required for:

- User ID
- PDF ID
- Topic ID
- Quiz Session ID
- Study Session ID
- Attempt ID
- Created Date

The exact indexes will be defined during database implementation.

---

# 19. Vector Search Performance

The vector search system should be optimized for efficient semantic retrieval.

The system should retrieve only the most relevant chunks required for the current AI task.

The system should avoid sending unnecessarily large amounts of PDF content to the AI model.

---

# 20. RAG Context Size

The RAG pipeline should control the amount of context sent to the AI model.

The system should:

- Retrieve relevant chunks.
- Rank retrieved chunks.
- Limit the final context size.
- Avoid unnecessary duplicate content.

This helps improve:

- AI response quality
- Response speed
- API usage
- Cost efficiency

---

# 21. Storage Scalability

Cloudflare R2 should be used to store user-uploaded files and other large binary objects.

The application database should not be used to store large PDF binary data directly.

The database should store metadata and references to stored files.

---

# 22. Database Backup

The production database should have a backup strategy.

Backups should protect against:

- Data Loss
- Database Failure
- Accidental Deletion
- Infrastructure Failure

The exact backup frequency will be determined during deployment planning.

---

# 23. Backup Verification

Backups should be periodically verified to ensure that they can be restored successfully.

A backup that cannot be restored should not be considered a reliable backup.

---

# 24. Data Recovery

The system should have a recovery process for critical infrastructure failures.

Recovery planning should consider:

- PostgreSQL
- Application Data
- PDF Metadata
- Study Data
- Quiz Data

Original PDFs should remain stored in Cloudflare R2 according to the configured storage and retention policy.

---

# 25. Availability

The production application should aim for high availability.

The system should minimize downtime caused by:

- Application Errors
- Service Restarts
- Temporary AI Provider Failures
- Background Job Failures

The exact uptime target will be finalized according to the client's infrastructure and hosting budget.

---

# 26. AI Provider Availability

The application should not completely fail if the Gemini API is temporarily unavailable.

The system should handle AI provider failures gracefully.

The user should receive a clear message.

Example:

> AI service is temporarily unavailable. Please try again later.

---

# 27. Service Independence

A temporary failure in one component should not unnecessarily bring down the entire application.

Examples:

If Gemini is unavailable:

- Dashboard should still work.
- Uploaded PDFs should still be visible.
- Study history should still be accessible.
- Quiz history should still be accessible.

If PDF processing is unavailable:

- Existing processed PDFs should remain accessible.

---

# 28. Maintainability

The application code should be organized into logical modules.

The architecture should separate:

- Frontend
- Backend
- AI Processing
- Database
- Storage
- Background Jobs

This should make future maintenance easier.

---

# 29. Code Quality

The system should follow consistent coding standards.

The codebase should use:

- Meaningful Names
- Clear Module Boundaries
- Reusable Components
- Proper Error Handling
- Environment-Based Configuration

---

# 30. Configuration Management

Environment-specific configuration should be separated.

Examples:

- Development
- Testing
- Production

Sensitive configuration should be provided through environment variables or secure secret management.

---

# 31. Environment Separation

The system should maintain separate environments where practical.

Recommended environments:

- Local Development
- Testing / Staging
- Production

Production credentials must not be used in local development.

- Production Database

- HTTPS

- Logging

# 35. Python AI Service Deployment

The Python AI Service should run as an independent service.

It should be able to:

- Receive AI processing requests.
- Process PDFs.
- Perform RAG operations.
- Communicate with Gemini.
- Return structured results.

# 36. Database Deployment

PostgreSQL should run in a production environment with:

- Persistent Storage
- Backup
- Access Control
- Secure Credentials

The database should not be publicly accessible without appropriate security controls.

---

# 37. Storage Deployment

Cloudflare R2 should store:

- Original PDFs
- Extracted Images
- Voice Recordings, if stored

Private files should remain protected.

---

# 38. HTTPS Requirement

Production application traffic must use HTTPS.

This includes communication between:

- Browser and ReactJS
- Browser and Laravel API

Internal service communication should also use secure communication mechanisms where required.

---

# 39. Monitoring

The production system should have monitoring for important services.

Monitoring should cover:

- Laravel Application
- Python AI Service
- Background Workers
- PostgreSQL
- Storage
- AI Provider Errors

---

# 40. Error Monitoring

The system should track:

- Application Exceptions
- API Failures
- AI Failures
- PDF Processing Failures
- Background Job Failures

The exact monitoring tool will be selected during technical architecture planning.

---

# 41. Health Checks

Production services should provide health checks where practical.

Health checks may verify:

- Application Availability
- Database Connectivity
- AI Service Availability
- Queue Worker Availability

A health check should not expose sensitive system information.

---

# 42. Logging and Observability

The system should maintain structured logs for troubleshooting.

Important logs should include:

- Timestamp
- Service
- Event Type
- Status
- Resource ID
- Error Type

Sensitive information must not be logged.

---

# 43. Security in Production

Production deployment must follow the security requirements defined in Part 6.

This includes:

- HTTPS
- Secure Authentication
- Authorization
- Protected API Keys
- Secure Database Access
- Private R2 Storage
- Input Validation
- Rate Limiting

---

# 44. Cost Efficiency

The system should be designed to minimize unnecessary AI usage.

The application should avoid:

- Reprocessing the same PDF unnecessarily.
- Regenerating identical embeddings unnecessarily.
- Sending entire PDFs to the AI model for every request.
- Making unnecessary duplicate AI requests.

The RAG pipeline should retrieve only relevant content.

---

# 45. Duplicate Processing Prevention

If a PDF has already been processed successfully, the system should avoid processing the same content again unnecessarily.

The system may use a PDF content hash such as SHA-256 to identify duplicate files.

---

# 46. Resource Management

The system should control resource usage for:

- CPU
- RAM
- Storage
- Database Connections
- AI API Requests
- Background Jobs

Resource limits should be configurable.

# 51. Data Consistency

The system should maintain consistent data between:

- ReactJS
- Laravel
- Python AI Service
- PostgreSQL
- pgvector
- Cloudflare R2

If an operation partially fails, the system should avoid leaving inconsistent records.

---

# 52. Transaction Handling

Database transactions should be used for operations where multiple related database changes must succeed together.

Examples:

- Final Quiz Submission
- Score Calculation
- Progress Update
- PDF Record Creation

---

# 53. Quiz Result Integrity

Once a quiz is finally submitted:

- The attempt should be locked.
- The score should be stored.
- The result should not change unexpectedly.

The final result should be reproducible from the stored quiz attempt data.

---

# 54. Study Progress Integrity

Study progress should be stored reliably.

If a user closes the application, the system should preserve the latest saved progress.

The user should be able to continue from the last saved state.

---

# 55. AI Output Reliability

AI-generated content should be grounded in the uploaded PDF wherever the operation requires PDF-specific knowledge.

The system should use the RAG pipeline to retrieve relevant PDF content.

The AI should not be treated as the sole source of truth for the uploaded document.

---

# 56. Source Traceability

AI-generated Study Mode explanations and relevant AI responses should provide source information where applicable.

The source information should allow the user to identify the relevant:

- PDF
- Section
- Page
- Topic

The exact source display format will be finalized during UI and Technical Architecture design.

# 58. Final Version 1 Acceptance Criteria

Version 1 will be considered functionally complete when a user can:

1. Create an account.

2. Log in.

3. Access their personal dashboard.

4. Upload an AME-related PDF.

5. Upload a PDF up to 100 MB.

6. Receive a clear message if the PDF exceeds 100 MB.

7. Split an oversized PDF into multiple parts and upload those parts separately.

8. See the processing status of the uploaded PDF.

9. Open a successfully processed PDF.

10. View the generated PDF context.

11. Select Study Mode.

12. Study the PDF topic-by-topic.

13. View AI-generated explanations based on the PDF.

14. View relevant source information where applicable.

15. Track Study Mode progress.

16. Track Study Mode time.

17. Select Quiz Mode.

18. Receive topic-by-topic quizzes.

19. Answer Single Choice MCQs.

20. Answer Multiple Choice MCQs.

21. Answer Text-based questions.

22. Answer Voice-based questions.

23. Receive at least 3 Text-based questions and 3 Voice-based questions in a complete quiz.

24. Attempt questions before final submission.

25. Change answers before final submission.

26. Skip questions.

27. Receive 0 marks for skipped questions.

28. Receive no negative marking.

29. Receive Easy, Medium and Hard questions according to the defined difficulty distribution.

30. Complete quizzes topic-by-topic.

31. Receive topic-level scores.

32. Receive an overall score.

33. Pass when the final score is 75% or higher.

34. View quiz history.

35. View study history.

36. View PDF-specific study time.

37. View overall study time.

38. View topic progress.

39. View quiz progress.

40. View final quiz results.

41. Continue progress after leaving and returning to the application.

42. Access only their own data.

43. Have their uploaded PDF and learning data protected.
